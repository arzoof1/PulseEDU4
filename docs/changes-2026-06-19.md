## What shipped — School Tours Phase 1

Three feature groups, all under School Tours (Enrollment Leads):

- **(A) Roadmap "Option A"** — schools can mark checkpoints "always include on
  every tour." The guide Tour Roadmap PDF now walks the **union** of the family's
  picks and the school's always-include highlights, in page order, with a badge
  on each stop (`★ Family requested`, `School highlight`, or both) plus a guide
  cue line for family picks. The public form shows always-include stops as a
  read-only "Always included on every tour" list (EN + ES).
- **(B) Assignment notification** — assigning a lead to a staff member now fires
  (after the DB commit, best-effort) an email to the new owner with the
  principals/admins CC'd, an optional SMS (gated by the school's SMS scope), and
  an in-app `admin_notifications` record.
- **(C) Tour Guide role + SMS scope** — a lightweight `capTourGuide` role that can
  be assigned a lead and open the Tour Roadmap for **their own** assigned leads,
  without the full tour-management surface. A new `tourSmsScope` school setting
  (`all` | `urgent`) gates assignment SMS (`urgent` = email only).

### DB schema (additive — auto-applied on boot)

These columns are created by `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` in
`seed.ts` (`ensureAdminHubSchema` + `ensureToursSchema`), so a deploy applies
them automatically. No manual migration needed.

- `school_settings.tour_sms_scope TEXT NOT NULL DEFAULT 'all'`
- `staff.cap_tour_guide BOOLEAN NOT NULL DEFAULT FALSE`
- `tour_checkpoints` are stored as JSON on the tour page row; the new
  `alwaysInclude?: boolean` field is part of that JSON payload (no column).

```ts
// Notify a staff member they've been made the owner of a tour lead. The
// principal/admins are CC'd for oversight. Best-effort, gated on email being
// enabled.
export async function sendLeadAssignedEmail(
  args: LeadAssignedEmailArgs,
): Promise<boolean> {
  if (!emailEnabled() || !args.to) return false;
  try {
    const { client, fromEmail } = await getUncachableResendClient();
    const rows = [
      ["Family", args.familyName],
      ["Phone", args.phone],
      ["Student(s)", args.childrenSummary],

            k,
          )}</td><td style="padding:4px 0; font-size:14px;"><strong>${escapeHtml(
            v,
          )}</strong></td></tr>`,
      )
      .join("");
    const body = `
      <p style="margin:0 0 14px 0;">Hi ${escapeHtml(args.assigneeName)},</p>
      <p style="margin:0 0 14px 0;">You've been assigned a tour at <strong>${escapeHtml(
        args.schoolName,
      )}</strong>. Please reach out to the family within one school day and print the Tour Roadmap before they arrive.</p>
      <table role="presentation" cellpadding="0" cellspacing="0" style="margin:0 0 20px 0;">${rows}</table>
      <p style="margin:0 0 22px 0; text-align:center;">
        <a href="${args.pipelineUrl}" style="display:inline-block; background:#0ea5a4; color:#ffffff; text-decoration:none; padding:12px 24px; border-radius:10px; font-weight:600; font-size:15px;">Open the lead</a>
      </p>`;
    const result = await client.emails.send({
      from: fromEmail,
      to: args.to,
      cc: args.cc.length ? args.cc : undefined,
      subject: `Tour assigned to you: ${args.familyName} — ${args.schoolName}`,
      html: shell("You've been assigned a tour", body),
      text: `Hi ${args.assigneeName},\n\nYou've been assigned a tour at ${args.schoolName}.\n\nFamily: ${args.familyName}\nPhone: ${args.phone}\nStudent(s): ${args.childrenSummary}\nAssigned by: ${args.assignedByName}\n\nOpen the lead: ${args.pipelineUrl}`,
    });
    if (result.error) throw new Error(result.error.message);
    return true;
  } catch (err) {
    logger.warn({ err }, "tour lead-assigned email failed");
    return false;
  }
}
```

```ts
// Allows full tour managers AND lightweight Tour Guides. Routes that use this
// must additionally enforce own-lead scoping for guides via canAccessLead().
function requireTourGuide(req: Request, res: Response, next: NextFunction) {
  const staff = (req as Request & { staff?: StaffRow }).staff;
  if (!staff || !canGuideTours(staff)) {
    res.status(403).json({ error: "Not authorized for School Tours" });
    return;
  }
  next();
}

// Full managers can touch any lead; a Tour Guide may only act on a lead they
// own. Returns true when the staff member is allowed to view/print this lead.
function canAccessLead(
  staff: StaffRow,
  lead: { assignedStaffId: number | null },
): boolean {
  if (canManageTours(staff)) return true;
  return lead.assignedStaffId === staff.id;
}

```

## What shipped — School Tours Phase 2 ("Never lose a lead")

Turns the visual overdue flag from Phase 1 into active automation. Five parts:

- **(A) "Still deciding" live stage.** `deciding` is now a real pipeline stage
  between `toured` and `closed` (not just a closing outcome). Moving a lead to
  Still Deciding stamps a business-day follow-up clock
  (`followUpDueAt = addBusinessDays(now, tourFollowUpBusinessDays)`); logging a
  contact on a deciding lead resets the clock and clears the escalation stamp.
- **(B) Stage-aware overdue + auto-archive.** The pipeline list, the lead
  drawer, and the escalation sweep all compute overdue through ONE shared helper
  (`overdueFor()` in `lib/tourReminders.ts`) so they can never disagree: a `new`
  lead past the first-contact window (`first_contact`), a `scheduled` tour whose
  time passed but was never logged (`tour_not_logged`), or a `deciding` lead past
  its follow-up clock (`follow_up`). Closed tours auto-archive after
  `tourArchiveDays` and drop off the default board (shown via `?view=archived`).
- **(C) Hourly escalation scheduler.** `runTourEscalations()` (node-cron, hourly,
  `TOUR_ESCALATION_CRON` override, gated `NODE_ENV!=="test"`) emails the lead
  owner — or the notify group when unassigned — CC'ing principal/admins, with
  reason-specific copy. Idempotent: at most one nudge per reason per 24h
  re-nudge window (`lastEscalatedReason` + `lastEscalatedAt`), re-sending
  immediately when the reason changes. Gated globally on
  `EMAIL_REMINDERS_ENABLED` and per-school on `tourEscalationEnabled`. Stamps an
  `escalation` event + an `admin_notifications` row (`tour_lead_overdue`) only
  after a successful send.
- **(D) Configurable SLA settings.** Admin "Follow-up & escalation" card in the
  Tours settings exposes first-contact hours, follow-up business days, archive
  days, and the escalation on/off toggle. Surfaced + persisted (clamped) on
  `GET`/`PUT /tours/page`; backed by new `school_settings` columns.
- **(E) Client board.** Still-deciding column, follow-up countdown badges + a
  drawer note, reason-aware overdue badges, and an Active/Archived tab.

Schema (additive `ALTER TABLE IF NOT EXISTS` in `seed.ts`, applied on boot —
drizzle-kit push is unavailable here): `tour_requests.follow_up_due_at`,
`closed_at`, `last_escalated_at`, `last_escalated_reason`;
`school_settings.tour_first_contact_hours`(24),
`tour_follow_up_business_days`(3), `tour_archive_days`(3),
`tour_escalation_enabled`(true). New `lib/businessDays.ts` (`addBusinessDays`,
weekend-skipping, no holidays).

No student IDs are rendered on any Phase 2 surface or in any escalation email
(tour leads are family/lead records and carry no FLEID).
